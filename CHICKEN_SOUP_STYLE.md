# AI 技术类文章写作风格指南

> **适用场景**：AI 工具评测、技术教程、效率工具对比、编程经验、Prompt 工程  
> **字数标准**：1500-2000 字，参考 [HUMANIZE_GUIDE.md](HUMANIZE_GUIDE.md) 降 AI 味规则  
> **核心原则**：真实测试数据 > 主观感受 > 通用结论；永远给代码/命令，不只给描述

---

## 🎯 核心写作原则

### 风格关键词
**实战、数据、诚实、可复现**

### 价值观
- 每一条结论背后都有真实测试（时间、结果、截图、命令）
- 承认工具的局限性，不只吹不踩
- 让读者能复现文章里的每一步操作
- 技术准确优先于表达流畅

---

## 📝 标题设计公式

```
标题 = 【具体工具/场景】+ 【量化结果/核心发现】+ 【吸引点】
```

### 高点击率标题类型

1. **对比式**
   - Cursor vs Claude Code：我用了一周，结论很意外
   - 我让 Kimi 和 Gemini 各自翻译了一本书，差距有点大

2. **实测数据式**
   - 我用 AI 写了 30 天日报：效率提升数据全记录
   - Claude Code 生成一个项目平均耗时：4 分钟（附完整记录）

3. **操作教程式**
   - 如何用 Claude Code 从零搭建一个完整博客（2025 最新版）
   - 5 分钟配置 Claude MCP：让 AI 直接读你本地文件

4. **反共识式**
   - 为什么我不推荐用 AI 写代码（大多数人用错了）
   - 主流 Agent 都翻车了：这 3 个场景目前没有好的方案

5. **清单式**
   - 2025 年我用过的 8 个 AI 编程工具，真实评测
   - Prompt 进阶的 7 个技巧，第 5 个大多数人不知道

### 标题禁区
- ❌ 纯功能罗列：「AI 工具的 10 大功能介绍」
- ❌ 太标题党：「AI 颠覆了一切！你必须知道！」（空洞）
- ❌ 没有具体性：「如何提升效率」（太泛）
- ✅ 始终带具体工具名、场景或数字

## 📐 文章结构模板

### 标准结构（1500-2000 字）

```
开篇钩子（150-250 字）
    ↓ 真实场景 / 反直觉发现 / 数据亮点
问题背景（300-400 字）
    ↓ 读者遇到的痛点，我之前也卡在这里
核心方案 / 工具对比（1000-1500 字）
    ↓ 含代码示例 / 命令 / 截图描述
真实测试结果（300-500 字）
    ↓ 含具体数字、时间、质量对比表
使用限制与注意事项（100-200 字）
    ↓ 诚实说它哪里不行
结论与行动建议（100-200 字）
    ↓ 我推荐你怎么做，谁适合/谁不适合
```

## ✍️ 开篇模板

**原则：上来就给读者一个"反直觉"或"可复现"的信息，不废话。**

```
开篇方式 1：反直觉数据
Claude Code 写代码的速度是 Cursor 的 1.8 倍，
但调试能力反而差了一截——这是我实测了 20 个项目后的结论。

开篇方式 2：真实场景
上周五晚上 11 点，我用 Claude MCP 让 AI
直接读了我本地 3000 行代码，花了 4 分钟生成了一个完整的数据导出脚本。
以前这件事我要手动做 2 小时。

开篇方式 3：直接痛点
写 Prompt 最浪费时间的不是想内容，
而是一次次调参等结果——直到我发现了一个 3 步模板，
把平均调试次数从 7 次降到了 2 次。
```

### 开篇禁忌
- ❌ 「随着 AI 技术的发展...」「在当今时代...」
- ❌ 上来就定义概念：「人工智能（AI）是指...」
- ❌ 长铺垫三段才进正题
- ✅ 第一句话就要有信息量或反直觉点

## 🎭 素材使用规则

### 必须有的素材类型

1. **代码片段**（60% 内容）
   - 必须可运行，不要截图只贴代码
   - 标注语言：`bash` / `python` / `typescript` 等
   - 关键行加注释说明

2. **命令/配置文件**（20% 内容）
   ```bash
   # 完整可运行的命令
   claude --print "Hello"
   ```

3. **对比数据表**（15% 内容）
   - 工具横评必须有量化指标
   - 时间、质量、成功率等可测量的维度

4. **个人判断**（5% 内容）
   - 诚实说哪个更好、哪个有坑
   - 不为了显得客观而两边各打 50 大板

### 素材来源优先级
1. 官方文档（首发 API 功能必须查官方）
2. 我的真实测试结果（最可信）
3. GitHub issues / Reddit / HN 热帖（社区反馈）
4. 第三方评测（需注明来源，交叉验证）

## 💎 文章节奏控制

### 段落节奏

```
快（1-2 句，抛结论）
    ↓
慢（3-5 句，给证据或代码）
    ↓
快（1-2 句，给下一个结论）
    ↓
慢（继续）
```

### 转折节奏

```
描述体验好的地方 → 诚实说哪里有坑
推荐 A → 明确说 B 哪里比 A 更好
说功能强大 → 补充具体的使用限制
```

## 🎬 结尾模板

```
【一句话总结核心结论】

适合读这篇文章的人：
- 想用 AI 辅助编程但不知道选哪个工具的
- 已经在用某个工具但想提效的

不适合的：
- 需要完全没有 AI 参与的专业代码审查
- 对 AI 有过度期待觉得能完全替代程序员的

下一步你可以：
1. [具体可操作的第一步]
2. [具体可操作的第二步]
```

**⚠️ 不要升华、不要喊口号、不要"让我们一起拥抱 AI"。**

**意疏AI口袋固定签名（每篇必加）：**
```
用 AI 让普通人变强。
用方法把复杂的事变简单。
用行动把人生带到更高的位置。
```

## 🎨 语言风格要求

### 句式特点
1. **短句为主**（60%）：每句 15-25 字，给结论
2. **代码注释穿插**（30%）：代码 + 一句说明
3. **长句**（10%）：仅用于解释复杂概念或给背景

### 技术准确性要求
- API 名、版本号、命令必须准确（发前核对官方文档）
- 数字必须真实（不要编"效率提升 300%"）
- 不确定的地方明确说「在我测试的版本中...」「目前最新版本已支持...」

### 诚实表达
```
❌ 「这个工具完美解决了所有问题」
✅ 「这个工具在 X 场景下表现很好，但在 Y 场景下还需要手动介入」

❌ 「比传统方式快 10 倍」（没给测试条件）
✅ 「重构同一个模块，用 Cursor 花了 8 分钟，手动写花了 35 分钟」

❌ 「这个 Prompt 技巧太牛了」
✅ 「这个结构化 Prompt 模板把模糊需求→可执行代码的转化率从 40% 提到了 72%」
```

### 段落要求
- **每段**：不超过手机屏幕 5 行
- **空行**：段落之间必须有空行
- **代码块**：独立成块，不要夹在长段文字中间

## 🔧 写作流程

### 步骤 1：确定核心问题
写之前先问自己：读者读完这篇文章能得到一个什么具体结论？
「Cursor 比 Claude Code 适合 X 场景」比「两款工具对比」好一万倍。

### 步骤 2：搭建测试框架
- 确定 2-3 个可量化的测试维度
- 每次只对比一个变量（如：同样需求，看谁的代码更简洁）
- 记录完整输出（代码、时间、主观质量评分）

### 步骤 3：搜官方文档
- 首发功能必须查官方更新日志
- API 参数以官方文档为准

### 步骤 4：写初稿
- 先给结论，再给证据
- 代码必须可运行
- 每给一个判断必须有具体数据支撑

### 步骤 5：降 AI 味
- 参考 [HUMANIZE_GUIDE.md](HUMANIZE_GUIDE.md) 第二遍审校规则
- 删除「显著提升」「深入了解」等书面词
- 加至少 5 处个人判断（「我觉得」「说实话」「我踩过的坑」）
- 朗读测试：听起来像在和工程师朋友聊，不是写论文

## ✅ 自检清单

### 事实准确性
- [ ] 所有数字都有测试依据？（不要编造）
- [ ] API 名、版本号、命令经过官方文档核对？
- [ ] 结论说清楚了适用范围和测试条件？

### 内容质量
- [ ] 核心判断在标题或前 200 字就说清楚了？
- [ ] 有至少 2 个可运行的代码/命令片段？
- [ ] 有对比数据表或量化结果？
- [ ] 诚实说了工具的局限性？

### 标题检查
- [ ] 带具体工具名或场景？
- [ ] 有量化信息（数字/时间/对比）？
- [ ] 不是纯功能罗列？

### 语言风格
- [ ] 没有「在当今时代」「随着...发展」等 AI 套话？
- [ ] 有 5+ 处个人判断/感受/踩坑记录？
- [ ] 读起来像和朋友聊天，不像写论文？

### 格式检查
- [ ] 代码块独立成块，语言标注正确？
- [ ] 段落不超过 5 行手机屏幕？

本指南是 **AI 技术类文章的专用写作规范**，在 SKILL.md 流程中被**自动调用**：

   - 第二遍风格审校：按 [HUMANIZE_GUIDE.md](HUMANIZE_GUIDE.md) 降 AI 味 + 本指南技术诚实性检查
